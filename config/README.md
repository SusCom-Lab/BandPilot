# config Directory Guide

Stores all runtime configurations, typically passed to `main.py` via `config/default_config.yaml`.

## Fields in default_config.yaml

```yaml
data:
  h100_training_data_path    # Bandwidth data path for H100 during training/model inference (default: 16M samples)
  h100_evaluation_data_path  # Bandwidth data path for H100 during real-data evaluation (default: 2GB samples)
  bandwidth_dict_dir         # Directory of per-node bandwidth dictionaries (pkl)
  model_save_dir             # Output directory for models and scalers during training
  evaluation_dir             # Output directory for evaluation results (MaxBW/Single_contention, etc.)

model:
  type                   # Currently only supports 'full' (BandwidthPredictor)
  hidden_dim / ...       # Transformer hyperparameters

training:
  enable_training         # Whether to train; if false, load existing model from default path
  batch_size, num_epochs, learning_rate, etc.  # Standard training params
  num_train_samples      # Number of training samples generated by dataset
  num_test_samples       # Number of random test samples for quick post-training checks

cluster:
  total_gpu              # Total GPUs in cluster (must be a multiple of 8)
  cluster_types          # List of cluster topology combinations
                         # e.g., ['H100_26H100_27H100_28H100_29', 'Het-4Mix']
                         # Het-4Mix means four 8-GPU servers (4090/A800/A6000/V100);
                         # cross-node bandwidth uses H100 CSV; final bandwidth takes the bottleneck of intra-/inter-node
  bw_switch              # Switch bandwidth type (propagated to evaluation filenames)

evaluation:
  enable_single_contention   # Whether to run single dispatch + background contention
  single_contention:
    repeat_num               # Repetitions per test_num
    if_dynamic               # true to sample available GPUs randomly; false means all GPUs are available
    contention_mode          # Contention modes; string or list ('common'/'intensive'/'idle')
    search_if_real_data      # Whether search uses real data directly
  max_bw_offline:
    enable                   # Whether to collect max_bw cache offline
    repeat_num               # Sample count (scalar or list)
    if_dynamic               # Same semantics as single_contention.if_dynamic
    contention_mode          # Same as single_contention.contention_mode; drives cache labels
    search_if_real_data      # Whether max_bw collection uses real data
    local_top_k              # Local candidates to keep per node
    max_combos_per_distribution  # Candidate cap per node distribution
    max_total_combos         # Global cap on evaluated combinations

random_seed / device     # Global RNG seed(s) (scalar or list) and training device
```

> To run the current “training + single-contention evaluation” flow:  
> 1. Set `model.type` to `'full'`.  
> 2. Enable `evaluation.max_bw_offline.enable: true` to collect offline `MaxBW_*.csv`.  
> 3. Set `evaluation.enable_single_contention` to `true` and configure `single_contention`.  
> 4. Run `python main.py --config config/default_config.yaml`; results will appear in `data.evaluation_dir/<cluster_type>/`.

## Data Path Configuration

### Split data paths for H100 experiments

For H100 experiments (configs where `cluster_types` contains `"H100"`), training and evaluation data can be separated:

- **Training/model inference (`h100_training_data_path`)**:
  - Used for model training and model-based decision making
  - Default: `Pune_H100_16M_binary.csv` (16M samples, same as training data)
  - Used when the algorithm decides with the model (`if_real_data=False`)

- **Real-data evaluation (`h100_evaluation_data_path`)**:
  - Used to evaluate decisions with real data
  - Default: `Pune_H100_2GB_binary.csv` (2GB samples, closer to real scenarios)
  - Used when the algorithm runs in real-data mode (`if_real_data=True`) or when evaluating decisions

### Data path selection logic

The system selects data paths as follows:

1. **Training**: always uses `h100_training_data_path`
2. **Model-based decision**:
   - If `if_real_data=False` (default), use `h100_training_data_path`
   - If `if_real_data=True`, use `h100_evaluation_data_path`
3. **Evaluation of decisions**:
   - Always uses `h100_evaluation_data_path` to reflect real scenarios

### Backward compatibility

- If only `h100_training_data_path` is provided and `h100_evaluation_data_path` is missing, evaluation will fall back to the training path.
- For non-H100 cluster types (e.g., `Het-4Mix`), the same CSV is used (controlled by the H100 path fields).

### Example

```yaml
data:
  # H100 experiment: split training and evaluation data
  h100_training_data_path: './Data/H100/Pune_H100_16M_binary.csv'
  h100_evaluation_data_path: './Data/H100/Pune_H100_2GB_binary.csv'
```
